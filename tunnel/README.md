# endgateway.cc tunnel

Lightweight Minecraft Java tunnel for small VPS deployments. This component exposes public TCP and UDP listeners on the VPS, forwards traffic through an outbound desktop agent tunnel, and optionally manages Cloudflare DNS records for active servers.

## Status

This repository is a buildable MVP, not the full production system described in the tunnel prompt yet.

Implemented now:
- VPS control plane with mTLS agent registration on `:8443`
- dynamic same-number TCP/UDP port allocation in `50000-60000`
- TCP gameplay forwarding over yamux streams
- UDP forwarding path for Simple Voice Chat
- Cloudflare DNS record create/delete on server start/stop
- desktop agent that opens the tunnel and rewrites `voicechat-server.properties`
- Dockerfiles for VPS and agent
- Docker Compose baseline for VPS deployment and optional agent testing
- GitHub Actions workflow that builds both images

Still missing for the full spec:
- HMAC request signing and replay protection
- rate limiting and per-scope auth on API endpoints
- heartbeat-driven stale session cleanup
- durable state storage beyond in-memory runtime state
- Docker deployment control endpoints
- structured audit persistence
- full admin API surface for registration, secret rotation, and DNS mutation workflows

That confirms the repository is publishable and container-buildable. It does not by itself prove full production readiness.

## Architecture

Components:
- `vps/`: Go control plane and public listeners
- `agent/`: Go desktop agent that maintains one outbound TLS connection

Runtime flow:
1. The desktop agent connects to `GET /tunnel` over mTLS.
2. The agent calls `POST /v1/servers/{id}/start`.
3. The VPS allocates one TCP port and one UDP port, using the same numeric port where possible.
4. The VPS opens listeners only for the active server.
5. The VPS optionally creates `mc-<id>.<domain>` in Cloudflare.
6. Minecraft TCP traffic and voice UDP traffic are tunneled to the desktop over yamux-backed streams.
7. `POST /v1/servers/{id}/stop` closes listeners, releases ports, and removes the DNS record.

## Current API

Implemented endpoints:
- `GET /healthz`
- `GET /tunnel`
- `POST /v1/servers/{id}/start`
- `POST /v1/servers/{id}/stop`
- `GET /v1/servers/{id}/allocation`
- `POST /v1/heartbeat`

Current auth model:
- mTLS for agent tunnel establishment
- no HMAC layer yet

Current response shape for `start` and `allocation`:

```json
{
  "tcp_public": 51234,
  "udp_public": 51234,
  "voice_host": "mc-srv1.example.com:51234",
  "dns_name": "mc-srv1.example.com"
}
```

## Security Model

Implemented:
- private CA based mTLS between agent and VPS
- no public port listeners in the managed range until a server is started
- Docker service hardening with `cap_drop: ALL` and `no-new-privileges`
- Cloudflare token injection through environment variables

Not implemented yet:
- request HMAC signatures
- nonce replay protection
- rate limits
- secret rotation workflows
- persisted audit log storage

## Docker

`docker-compose.yml` includes:
- `vps-tunnel`: default VPS service
- `desktop-agent`: optional profile for agent-side testing

Prepare:

```bash
cd tunnel
cp .env.example .env
mkdir -p certs data agent-data
```

Start the VPS:

```bash
docker compose up -d --build vps-tunnel
```

Start the agent profile locally for testing:

```bash
docker compose --profile agent up -d --build desktop-agent
```

## Certificates

Example local certificate generation:

```bash
openssl req -x509 -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 -days 3650 -nodes -keyout certs/ca.key -out certs/ca.crt -subj "/CN=tunnel-ca"
openssl req -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 -nodes -keyout certs/server.key -out certs/server.csr -subj "/CN=vps"
openssl x509 -req -in certs/server.csr -CA certs/ca.crt -CAkey certs/ca.key -out certs/server.crt -days 365
openssl req -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 -nodes -keyout certs/agent.key -out certs/agent.csr -subj "/CN=srv1"
openssl x509 -req -in certs/agent.csr -CA certs/ca.crt -CAkey certs/ca.key -out certs/agent.crt -days 365
```

## Minecraft Backend Notes

Recommended backend settings:

`server.properties`
```properties
server-ip=127.0.0.1
server-port=25565
enable-query=false
online-mode=true
```

`paper.yml`
```yaml
proxies:
  proxy-protocol: true
```

Generated by the agent in its working directory:

`voicechat-server.properties`
```properties
port=24454
voice_host=mc-srv1.example.com:51234
bind_address=127.0.0.1
```
